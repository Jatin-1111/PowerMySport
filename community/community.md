# Community

## What This Project Does
The `community` project is the social hub for PowerMySport. It allows users to ask questions (Q&A), discover and join groups based on their interests, follow topics/sports, and communicate directly through real-time Direct Messages (DMs) and Group Chats.

## Local Setup
1. **Prerequisites**: Node.js, `server` project, and Redis must be running locally.
2. **Install Dependencies**: `npm install` inside the `community` folder.
3. **Environment Setup**: Copy `.env.example` to `.env.local` (or `.env`).
4. **Run Server**: `npm run dev` starts the community frontend.

**Required Env Variables**:
- `NEXT_PUBLIC_API_URL`: Backend API URL (e.g., `http://localhost:5000/api`).
- `NEXT_PUBLIC_SOCKET_URL`: The Socket.IO server URL (derived from API URL if left blank).
- `NEXT_PUBLIC_MAIN_APP_URL`: The URL of the `client` app (e.g., `http://localhost:3000`) for login redirects.
- `NEXT_PUBLIC_CHAT_BUCKET_DOMAIN`: Amazon S3 bucket domain for chat image uploads.

## Key Data Flows

### Real-Time Chat Message Flow
1. **Send**: User types a message in `CommunityChatPanel.tsx` and hits send.
2. **Optimistic Update**: Message is immediately added to the UI and cached via `chatDB.upsertCachedMessage`.
3. **API Call**: `communityService.sendMessage` is called to persist it to the backend.
4. **Socket Broadcast**: The backend persists it and broadcasts a Socket.IO event (`new_message`) to all users in the chat room.
5. **Receive**: `CommunityChatPanel` listens for `new_message`, updates state, and caches the message in IndexedDB.

### Q&A Post Creation
1. **Initiate**: User clicks "Ask Question" triggering `AskQuestionModal.tsx`.
2. **Submit**: Form data (title, body, tags, sport, city) is sent via `communityService.createCommunityPost`.
3. **Update**: Upon success, `QnAFeedClient.tsx` re-fetches or optimistically prepends the new post to the feed.

## State Management
- **Local State**: Managed with React `useState` and custom hooks like `useConversationState` and `useCommunityPage`.
- **Global / Real-time State**: Managed by Socket.IO event listeners updating local states. 
- **Offline Cache**: `chatDB.ts` acts as an offline-first storage mechanism for chat history, mitigating slow networks and reducing DB queries.

## API & External Services
- **Backend API**: Uses `communityService.ts` via an Axios instance (`src/api/axios.ts`) to talk to the `server`.
- **WebSockets**: Connects to the `/community` namespace on the `server` using `socket.io-client`.
- **AWS S3**: Handles chat image uploads directly from the browser using presigned URLs generated by the backend (`getChatImageUploadUrl`).

## Routes
- `/` - Community Landing Page
- `/chats` - Active conversations and DMs
- `/discover` - Find new groups and users
- `/q` - Questions and Answers feed
- `/q/[postId]` - Detailed view of a Q&A post
- `/members/[userId]` - View a community member's profile
- `/following` - Manage followed groups and topics

## How It Talks to Other Projects
The `community` project relies entirely on the `server` project for data and real-time Socket.IO communication. 
It integrates loosely with the `client` project: unauthenticated users attempting to interact are redirected to the `client`'s login page using `NEXT_PUBLIC_MAIN_APP_URL`. The session cookie set by `server` is shared across subdomains, allowing seamless authenticated transitions between `client` and `community`.

## Naming & Code Conventions
- Uses Next.js App Router conventions.
- Custom hooks (e.g., `useDebouncedSearch`, `useAsync`) encapsulate complex logic and are centrally exported from `src/hooks`.
- IndexedDB operations are abstracted in `src/db/chatDB.ts`.

## Known Gotchas
- **Socket Disconnections**: If the Redis server on the backend drops, the Socket.IO server falls back to single-instance mode. Messages might not sync across multiple backend instances.
- **Login Redirects**: Because `community` lacks its own auth UI, the `redirect.ts` utility relies on `NEXT_PUBLIC_MAIN_APP_URL`. If this is misconfigured locally, login loops or dead-ends will occur.
- **IndexedDB Stale Data**: `chatDB` caches messages to make the app feel instant. If the local IndexedDB diverges significantly from the server, users might see outdated messages until a hard refresh.

## 404 / Not Found
A custom `src/app/not-found.tsx` renders for unmatched routes, shown inside the community shell (top nav) as a centred card with "Community home" and "Browse Q&A" links.
